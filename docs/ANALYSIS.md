# Candidate Search System — Submission Analysis

## Direction chosen

I built **job search (find candidates)**: given a hiring-manager style natural-language query, return the most relevant **candidate profiles**.

Why this direction (and why I did not index jobs in this submission):
- With a 24-hour constraint, I prioritized a complete, demo-ready system with transparent ranking and explanations on the smaller corpus (~2k candidates), which makes iteration and debugging much faster.
- The jobs dataset is much larger (~15k+) and would require additional work to normalize fields and tune ranking for salary/contract filters; doing both well would reduce quality.
- The architecture is corpus-agnostic: adding job search is straightforward by swapping the loader to read job rows, building a `combined_text` for jobs, and creating a second FAISS index + reranker.

## Approach (what / how / why)

### High-level idea
- Use **semantic retrieval** to handle vague queries and varied phrasing.
- Layer a lightweight **intent extractor** to capture explicit constraints (YOE, location, stability).
- Apply a simple, explainable **hybrid reranker** to ensure hard-ish constraints influence ordering.
- Provide a **per-result breakdown** showing exactly how each result was scored.

### System flow

```
User query
  |
  v
Intent extraction
  - Gemini (if configured) OR regex fallback
  - Output: {skills, min_yoe, location, stability_required, is_startup}
  |
  v
Semantic retrieval (FAISS)
  - Embed query with sentence-transformers
  - Retrieve top K by vector similarity
  |
  v
Hybrid reranking
  - Convert distance -> base_score
  - Apply multipliers for constraints (YOE/location/stability)
  - Output: final_score + breakdown flags
  |
  v
Explanation generation
  - Deterministic ranking breakdown (always)
  - Optional Gemini narrative (2–3 sentences)
  |
  v
Top-N results (CLI: top 20)
```

## Alternatives considered

- Pure keyword filtering (BM25): fast and interpretable, but struggles with vague queries and synonymy in resumes.
- Pure semantic search only: good recall, but often violates constraints like “4+ years” or “not a job hopper”.
- Hard filtering (drop candidates not meeting constraints): too brittle because the data is partially unstructured; constraints can be missing/implicit.

Hybrid retrieval + reranking was chosen for a practical quality/complexity trade-off.

## How ranking works (detailed)

1) **Base semantic score**
- Encode each candidate’s `combined_text` with `sentence-transformers/all-MiniLM-L6-v2`.
- Search using FAISS `IndexFlatL2`.
- Convert L2 distance to a score:
  - `base_score = 1 / (1 + distance)`

2) **Intent-aware multipliers** (applied in reranking)
- If extracted `min_yoe` and candidate `total_years_exp >= min_yoe`: `* 1.4`
- If extracted `location` is a substring match in candidate text: `* 1.3`
- If `stability_required`:
  - if `is_job_hopper == False`: `* 1.5`
  - else: `* 0.5`

3) **Explanation / breakdown per result**
Each result prints:
- base_score
- multiplier and which boosts/penalties applied
- final_score
- extracted intent summary
- plus a short narrative explanation (Gemini if available, fallback otherwise)

## Where it breaks (2–3 failure cases)

1) **Location normalization / ambiguity**
- Example failure: query says "Bangalore" but resume says "Bengaluru" or "BLR".
- Why: reranker currently uses a substring match for location.
- Fix: build a location normalizer / alias map + city extraction from resume text.

2) **Skill extraction recall**
- Example failure: query requires a niche skill but it appears in resume as a synonym or buried in a project.
- Why: intent extraction is best-effort and `combined_text` may not surface all relevant skills consistently.
- Fix: add explicit skill extraction (keyword/NER) and store normalized skill lists, then rerank with skill overlap.

3) **Experience estimation errors from unstructured dates**
- Example failure: education timelines or unusual date formats get parsed as work experience.
- Why: date-range parsing is heuristic.
- Fix: better resume section segmentation (Work vs Education), more date patterns, and confidence scoring.
